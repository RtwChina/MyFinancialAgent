## Context

项目已经完成账户化操作计划改造：`investment_accounts` 保存账户名称、币种、总资产、可用资金等基础信息，`account_live_action_plans` 保存当前活态操作计划，`daily_review_action_plans` 保存复盘日操作计划快照。后续 migration 已为活态计划和复盘日计划增加 `position_amount`，前端在个股编辑中以“万”为输入单位，保存时转成原始货币单位。

用户现在需要一个统计汇总视图，用来按 USD/CNY 分开看账户资金和标的资金分布。这个视图不是券商级真实持仓报表，而是基于当前手工维护的操作计划和账户资金字段生成的资金结构估算。

相关约束：

- 运行时必须保持 `APP_ENV` 环境标识，测试环境和生产环境使用各自 D1/配置。
- 第一版不引入第三方券商接口或实时汇率接口，因此没有不可控外部数据依赖；美元人民币参考值使用固定汇率 `6.80`。
- 第一版以当前活态计划为统计来源，历史复盘日视图留作后续扩展。

## Goals / Non-Goals

**Goals:**

- 提供一个资金分布汇总视图，按币种分组展示账户总资产、估算持仓和未分配金额。
- 汇总同币种内每个标的的总金额，并展示该标的在各账户中的金额和账户内占比。
- 展示每个标的占该币种账户总资产的比例，公式为 `标的同币种合计金额 / 同币种账户总资产`。
- 对缺失 `position_amount` 的计划，使用 `current_position` 仓位区间中位数估算金额。
- 对估算金额显示 `*`，tooltip 展示估算规则。
- 保持金额单位统一：DB/API 使用原始货币单位，页面输入和展示使用“万”。
- 保持统计页为轻量读取能力，不改变操作计划保存语义。

**Non-Goals:**

- 不做券商或基金平台同步。
- 不做真实收益、成本、成交、税费、资金流水或盈亏归因。
- 不做 USD/CNY 跨币种总计；美元分组摘要允许按固定 `1 USD = 6.80 CNY` 显示人民币参考值。
- 不解析自由文本里的金额。
- 不在第一版支持历史复盘日资金分布对比。

## Decisions

### Decision 1: 汇总数据以 Worker API 统一计算

资金分布汇总应由 Worker 提供统一读取接口，例如 `GET /api/account-fund-summary?scope=current`。前端只负责展示，不重复实现核心汇总逻辑。

理由：

- 汇总逻辑涉及账户、活态计划、仓位区间、金额单位和异常降级，放在后端更容易测试和复用。
- 后续如果要支持复盘日历史汇总，可以扩展同一套计算器输入源。
- 前端避免出现与后端不一致的金额估算。

备选方案是前端直接组合 `/api/investment-accounts` 和 `/api/account-live-action-plans`。这个方案实现快，但会把业务规则散在页面层，后续维护成本更高。

### Decision 2: 第一版只统计当前活态计划

第一版输入源为：

- `investment_accounts`：启用账户、币种、总资产、排序。
- `account_live_action_plans`：当前活态操作计划、标的、账户、当前仓位、手填金额。

不读取 `daily_review_action_plans` 作为默认来源。

理由：

- 用户当前要解决的是“当前账户整体资金结构看不清”。
- 历史复盘日需要处理账户快照、当日计划、之后账户改名或资金变化等问题，应该单独设计。

### Decision 3: 金额估算规则手填优先，仓位中位数兜底

每条计划的统计金额按以下顺序计算：

1. `position_amount` 为有效非负数时，使用该值，来源标记为 `exact`。
2. 否则，按账户 `total_assets × current_position 中位数` 估算，来源标记为 `estimated`。
3. `current_position` 为 `0%` 或动作属于准备开仓/已清仓类时，金额为 0，并从资金分布汇总标的矩阵中隐藏。
4. 账户缺少有效 `total_assets` 时，该计划不能参与金额估算，来源标记为 `unavailable`，页面显示 `-`。

仓位中位数映射：

| current_position | 中位数 |
| --- | --- |
| `0%` | 0 |
| `0-5%` / `0%-5%` | 2.5% |
| `5%-10%` | 7.5% |
| `10%-15%` | 12.5% |
| `15%-20%` | 17.5% |
| `20%-25%` | 22.5% |
| `25%-30%` | 27.5% |
| `>30%` | 30% |

`>30%` 是开放区间，第一版按 30% 保守估算，避免统计页夸大集中度。

### Decision 4: 按币种隔离汇总，不做跨币种合计

响应结构应按 `currency` 分组，例如 `groups: [{ currency: "USD" }, { currency: "CNY" }]`。每个分组独立计算总资产、估算持仓、未分配金额、账户列表和标的列表。

理由：

- USD 和 CNY 不应在没有明确汇率日期的情况下合并。
- 用户当前表达的是“美元和人民币账户汇总统计”，不是换算成单一基准货币。

美元分组中的摘要金额、账户覆盖金额和标的总金额均可附带灰色人民币参考值；固定汇率 `1 USD = 6.80 CNY` 只在“美元账户”标题旁的提示图标中解释，不重复写在每个数字后。参考值不参与 CNY 分组、不影响占比计算，也不构成跨币种合计。

### Decision 5: UI 使用资金矩阵而非来源说明表

页面结构参考 `tests/summary-demo.html`：

- 汇总入口放在“账户管理”页面标题操作区，不占用顶层导航；点击后用浏览器原生新标签打开 `?view=fundSummary`，原账户管理页面保留，汇总页顶层仍标示“账户管理”作为归属。
- 顶部标题为“资金分布汇总”。
- 每个币种一个完整分组。
- 分组头部显示总资产、估算持仓、未分配。
- 账户覆盖以横向账户带展示在摘要头部下方，避免侧栏压缩矩阵宽度；账户较多时账户带自身可横向浏览。
- 矩阵表在账户带下方占据完整内容宽度，展示顺序为：标的、每个账户列、总金额、占总资产。
- 汇总字段放在账户列右侧，使用户先比较账户分布，再读取币种内的总金额和“占总资产”比例。
- 不展示“备注 / 来源”列；估算来源通过金额后的 `*` 和 tooltip 表达。

样式应延续账户管理和复盘列表现有的浅色工作台语言：紧凑表头、轻边框、克制颜色与清晰数值层级，不使用突出的卡片阴影。桌面端应避免页面级横向溢出；账户列较多时，仅账户带和矩阵区域各自在容器内横向滚动。移动端可纵向堆叠币种摘要和账户带，矩阵保持横向滚动。

金额来源的视觉提示只由估算金额后的 `*` 及 tooltip 承担。手填金额不使用单独颜色或图例，与普通金额使用相同的淡蓝进度表达；汇总分组和进度的强调色采用低饱和淡蓝，避免灰沉或绿色被误读为涨跌/好坏状态。

最终确认配色采用 B 方案：摘要头部保持白底；`账户覆盖` 行不整体铺色，仅各账户卡片使用浅蓝底突出归属；矩阵表头与持仓横条延续同一淡蓝层级。

为保证浅蓝账户卡上的占比仍可快速识别，账户覆盖横条和标的横条统一使用浅蓝灰空轨道 `#e2edf8` 与中等蓝填充 `#5a95cd`，不因金额来源变化颜色。

### Decision 6: APP_ENV 与测试隔离

新增 API、测试数据和 smoke 测试必须遵循现有环境隔离：

- 本地/测试环境使用测试 D1 和测试端口。
- 生产环境读取生产 D1。
- 不在测试中写入生产账户或生产活态计划。
- Smoke 测试应通过 `APP_ENV` 或现有 wrangler/test 配置确认目标环境。

## Risks / Trade-offs

- [Risk] 估算金额可能被误认为真实持仓金额。  
  → Mitigation: 金额后显示 `*`，tooltip 明确“按仓位区间中位数估算”；页面副标题说明这是基于操作计划的资金分布。

- [Risk] 账户总资产单位不一致会导致汇总错误。  
  → Mitigation: 实施前先统一账户总资产表单单位，测试覆盖输入 `2` 展示 `$2万` 且 DB 保存 `20000`。

- [Risk] `position_amount` 为空但 `current_position` 不规范。  
  → Mitigation: 复用/集中仓位归一化逻辑，无法识别时不参与估算并显示 `-`。

- [Risk] 同一 symbol 在不同币种账户里出现时误合并。  
  → Mitigation: 先按 `currency` 分组，再在分组内部按 symbol 汇总。

- [Risk] 账户列增加后表格过宽。  
  → Mitigation: 表格容器横向滚动，页面主体不横向溢出；列宽使用稳定约束。

## Migration Plan

1. 确认 `024_action_plan_position_amount.sql` 已在目标环境应用。
2. 修复并验证账户总资产/可用资金的万元输入与原始单位存储一致性。
3. 新增后端汇总计算器和读取 API，先在本地/test 环境验证。
4. 在账户管理流程内新增统计汇总入口和页面。
5. 增加 smoke/API 测试覆盖汇总计算和页面展示。
6. 部署到生产前运行本地 smoke，并确认 `APP_ENV` 指向正确环境。

Rollback:

- 该功能第一版主要是新增读取 API 和页面入口，不改变现有保存语义。若上线异常，可隐藏入口或回滚静态资源；数据层无需回滚。
