## Context

当前 `collect-prices` 主任务在北京时间清晨运行，直接依赖 Yahoo Finance 获取最近一根已收盘的价格数据。线上日志已经出现过这种情况：

- 日志显示主任务曾拿到 `价格: None`
- 同日稍后重跑时，Yahoo 又能返回正常价格

这说明问题更像是 Yahoo 的短时不稳定，而不是持续缺少该标的数据。与其改造主任务的市场路由和多源 fallback，当前更适合增加一个延迟补偿任务，专门修复已落库的空价格记录。

## Goals / Non-Goals

**Goals:**
- 在北京时间 `08:00` 自动重试修复最近 `3` 天的空价格记录
- 只修复 `k_date` 已明确、`current_price` 为空的记录
- 修复时以记录自身的 `k_date` 为准，避免误用“最新一根价格”
- 只使用 Yahoo，保持第一版范围克制

**Non-Goals:**
- 不改主任务 `collect-prices` 的取价逻辑
- 不引入 AKShare / Finnhub / Tushare 等备用源
- 不修复 `k_date` 为空的 placeholder 记录
- 不处理超过最近 `3` 天的历史坏记录

## Decisions

### 1. 以“坏记录修复”而不是“主链路 fallback”为中心

修复任务聚焦于已经写入 `stock_raw` 的坏记录，而不是在主链路运行时判断是否切备用源。

这样做的原因：
- 更贴近当前真实问题：Yahoo 短时异常后会恢复
- 不需要重写主任务的市场判断与交易日逻辑
- 能直接修复数据库中已经存在的空值行

### 2. 候选范围限制为最近 3 天且 `k_date` 已存在

修复任务只扫描：

- `k_date IS NOT NULL`
- `current_price IS NULL`
- `k_date` 位于最近 `3` 天窗口内

这样可以保证：
- 修复对象是“已知目标日期”的坏记录
- 避免重新推断目标交易日
- 避免扫过多历史垃圾数据

### 3. 修复查询必须以记录自身 `k_date` 为准

修复任务不能重跑“取最新一根”的主逻辑。对于每条坏记录，必须显式验证 Yahoo 返回的数据是否命中该记录自身的 `k_date`。

**修复成功条件：**
- Yahoo 返回目标 `k_date` 的记录
- 该记录 `Close` 非空

**修复失败条件：**
- Yahoo 查不到目标 `k_date`
- 或目标 `k_date` 的 `Close` 仍为空

### 4. 修复写入使用 update 语义

修复任务不是插入新记录，而是修复已有记录。因此写入必须按 `(symbol, k_date)` 更新已有行，而不是复用 `INSERT OR IGNORE`。

建议更新字段包括：
- `current_price`
- `change_percent`
- `volume`
- `captured_at`

### 5. 新增独立调度，不复用主任务 rerun

修复任务应作为独立的 GitHub Action / 入口运行于北京时间 `08:00`，而不是在主任务失败时自动重跑整个 `collect-prices`。

这样可以：
- 将“主采集”和“延迟修复”解耦
- 避免整批重跑
- 单独观察修复成功率

## Risks / Trade-offs

- **[中风险] Yahoo 08:00 仍然异常**：修复任务可能依然失败，但日志会帮助量化问题是否只是短时抖动
- **[低风险] 最近 3 天窗口可能漏掉更老坏记录**：这是有意收缩范围；如后续确认有需要，可再扩大窗口
- **[低风险] 仅修复 `current_price` 为空的记录**：不会覆盖“日期错误但价格非空”的问题，这属于后续更严格治理范围
