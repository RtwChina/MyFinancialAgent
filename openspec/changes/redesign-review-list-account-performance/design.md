# 设计说明

## 当前形态

现有复盘列表后端返回的 `action_plan_summary` 主要是把操作计划标的拼接成字符串，前端再放到 `操作计划` 列里展示。同时列表还有单独的 `状态` 列，以及每行的操作按钮。

在操作计划已经账户化之后，这个表达方式不够贴合真实使用场景。用户关心的是：

```text
老虎-美股        USD 3.2 万
东方财富-国内    CNY 26 万
天天基金-国内    CNY 17 万
```

所以列表应该优先总结“这一天这些账户受到什么影响”，而不是重复一串标的。

## 推荐行结构

每一行围绕四件事组织：

```text
复盘日        账户表现 / 估算影响         当日主线           操作
2026-05-15   老虎-美股  1~2%            AI 主线摘要        查看  编辑
             东方财富  -1~0%
             天天基金   ≈0%
```

状态合并进日期单元格：

```text
2026-05-15
未复盘
```

已完成行不展示 `已复盘` 或 `已完成`，没有异常提示本身就代表它是正常完成态。

## 复盘日提示

`复盘日` 表头提供提示，用来解释日期口径：

- 复盘日代表美国交易日日期。
- 对美股正常交易时段来说，北京时间大致是该日期晚上到次日凌晨。
- 列表行里不再重复展示具体北京时间窗口。

如果现有前端或后端已经有类似 `buildReviewCycle(archiveDate)` 的时间窗口计算逻辑，实现时应复用它来生成更准确的提示示例。若没有，就保持提示为通用解释，避免写死误导性的精确时间。

## 当日主线来源

`当日主线` 不能使用 demo 里的硬编码文案。建议来源优先级：

1. 已复盘后的总结字段，例如 `trading_summary`。
2. `market_sentiment` 和 `sector_rotation` 的首句组合。
3. 可用的 AI 分析字段，例如 `daily_major_events` 或 `linkage_logic_chain`。
4. 都没有时显示 `待复盘`。

## 复盘日账户快照

复盘完成后，按复盘日记录一条账户快照。该快照内部使用结构化数据保存多个账户的复盘后确认值。

```text
一个复盘日 = 一条账户快照记录
一条账户快照记录内包含多个账户结果
```

例如 2026-05-15 复盘完成后，保存一条：

```text
archive_date = 2026-05-15
accounts = [
  老虎-美股,
  东方财富-国内,
  天天基金-国内
]
```

这样账户当前总金额之后继续变化，也不会影响历史复盘日的列表展示。

建议表结构：

```sql
CREATE TABLE IF NOT EXISTS review_account_snapshots (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    archive_date TEXT NOT NULL UNIQUE,
    accounts_snapshot TEXT NOT NULL,
    snapshot_source TEXT NOT NULL DEFAULT 'manual_review',
    notes TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

`accounts_snapshot` 保存结构化 JSON：

```json
[
  {
    "accountId": 1,
    "accountName": "老虎-美股",
    "currency": "USD",
    "totalAssets": 3.2,
    "availableCash": 0,
    "netCashFlow": 0,
    "dailyPnl": null,
    "dailyPnlPercent": null,
    "impact": {
      "source": "estimated_action_plan",
      "valuePercent": 0.72,
      "rangeLabel": "0~1%",
      "direction": "gain",
      "contributors": 3,
      "skippedReasons": []
    },
    "notes": ""
  },
  {
    "accountId": 2,
    "accountName": "东方财富-国内",
    "currency": "CNY",
    "totalAssets": 26,
    "availableCash": null,
    "netCashFlow": 0,
    "dailyPnl": null,
    "dailyPnlPercent": null,
    "notes": ""
  }
]
```

字段口径：

- `archive_date`：复盘日，一天一条。
- `accounts_snapshot`：该复盘日所有账户的结构化快照。
- `accountId`：账户 id，用于关联账户管理页。
- `accountName`：保存当时账户名，避免未来账户改名后历史显示变化。
- `currency`：快照币种，默认取账户币种。
- `totalAssets`：复盘完成后确认的账户总资产。
- `availableCash`：复盘完成后确认的可用资金。
- `netCashFlow`：当日净入金/出金。入金为正，出金为负。
- `dailyPnl`：当日真实盈亏，可由用户录入，也可由相邻复盘日快照计算后保存。
- `dailyPnlPercent`：当日真实收益率。
- `snapshot_source`：整条快照来源，例如 `manual_review`、`imported_statement`、`calculated`。

## 账户影响计算口径

本变更中“账户表现 / 账户影响”的核心口径以现有代码中的操作计划估算为准，不使用账户总资产差额倒推。

```text
单标的账户影响百分比 =
  当前仓位中位数 * 标的复盘日涨跌幅

账户影响百分比 =
  sum(同账户内可纳入计算的单标的账户影响百分比)
```

这里的仓位来自结构化操作计划的 `currentPosition`，例如 `15%-20%` 取中位数 `17.5%`，`0%` 取 `0%`。标的涨跌幅来自复盘日行情 `stock_raw.change_percent`。

如果同账户内某个标的是该复盘日第一天出现在操作计划中，则该标的当天不纳入账户影响，原因标记为 `新增标的，当日不计入`。如果缺少可解析仓位或缺少复盘日涨跌幅，则该标的不纳入计算，原因标记为 `缺少仓位或价格涨跌幅`。

`dailyPnl` 和 `dailyPnlPercent` 保留为手动确认或未来外部对账字段，不作为本次详情页账户影响展示的计算来源。系统不得把 `totalAssets`、上一复盘日 `totalAssets` 和 `netCashFlow` 的差额自动当成账户影响。

## 复盘完成时的写入流程

点击完成复盘时，系统应展示或保存该复盘日的账户快照信息：

- 默认带出账户管理页中的当前 `total_assets` 和 `available_cash` 作为初始值。
- 用户可以在完成复盘前调整每个账户的总资产、可用资金、净入金/出金和备注。
- 保存完成时按 `archive_date` upsert 一条复盘日账户快照，并将完成当刻按操作计划与行情计算出的账户影响固化到对应账户条目。
- 保存快照后，可以同步更新账户管理页的当前 `total_assets` 和 `available_cash`，让账户管理页保持最新值。

这样列表页读取历史时使用 `review_account_snapshots.accounts_snapshot`，账户管理页展示当前值，两者不会互相污染。

## 复盘详情页账户影响

复盘详情页和复盘过程中应展示与列表同口径的账户影响，但展示粒度可以更完整。

### 复盘过程中的实时账户影响

用户编辑操作计划时，详情页应根据当前操作计划、账户归属、当前仓位和复盘日行情实时生成账户影响预览。该预览应复用后端账户影响计算逻辑，避免前端和列表产生两套公式。

实时预览可以展示：

- 账户名称和币种。
- `账户估算影响` 标签。
- 精确 `valuePercent` 或紧凑区间 `rangeLabel`，详情页允许比列表更具体。
- 参与计算的标的数量 `contributors`。
- 跳过原因，例如 `新增标的，当日不计入`、`缺少仓位或价格涨跌幅`。

### 完成复盘后的固化账户影响

点击完成复盘时，系统应使用同一套账户影响计算逻辑生成结果，并将结果保存到 `review_account_snapshots.accounts_snapshot[].impact`。完成后再次打开复盘详情页时，系统优先展示已保存的 `impact`，不得因为之后修改操作计划、修复行情或调整账户资金字段而让历史详情页静默漂移。

如果历史复盘没有固化 `impact`，但仍存在账户化操作计划和复盘日行情，详情页可以提供“按当前数据重算”的估算展示，但必须标记为未固化当前估算。

## 账户估算影响与复盘结果口径

列表支持三种可信度层级。

### 手动或外部对账盈亏

如果该复盘日存在账户每日快照，并且用户或外部对账流程明确提供了 `daily_pnl` 或 `daily_pnl_percent`，系统可以展示为真实账户盈亏。本次“复盘详情页账户影响”不使用这组字段作为主要计算来源，也不得用账户总资产差额自动生成它们。

### 已记录快照但暂无盈亏

如果该复盘日存在账户快照，但缺少上一交易日快照，也没有手动录入盈亏，则展示账户已记录状态，例如：

```text
老虎-美股        已记录快照
东方财富-国内    已记录快照
天天基金-国内    已记录快照
```

列表不应把当前总资产变化误认为当日盈亏。

### 账户估算影响

如果没有已固化的账户影响，可以根据账户化操作计划、当前仓位和复盘日价格涨跌生成 `账户估算影响`。

```text
单标的账户影响百分比 =
  当前仓位中位数 * 标的复盘日涨跌幅

账户影响百分比 =
  sum(可纳入计算的单标的账户影响百分比)
```

示例：

- `15%-20%` 取中位数 `17.5%`
- `5%-10%` 取中位数 `7.5%`
- `0%` 取 `0%`
- 未知或无法解析的仓位不参与估算

示例：

```text
当前仓位 = 15%-20%，取 17.5%
标的复盘日涨跌幅 = +4%

单标的账户影响 = 17.5% * 4% = 0.7%
```

### 新增标的排除规则

如果某个标的是该复盘日第一天新增到该账户的操作计划中，则该标的当天不纳入估算影响。

原因是无法确认该标的是盘前买入、盘中买入、盘后买入，还是只是复盘后加入观察。如果把当天 K 线涨跌直接算进去，会把不确定的交易时点伪装成确定收益。

建议判定方式：

- 找到当前复盘日之前最近一个存在操作计划的复盘日。
- 如果同一 `accountId + symbol` 在上一份操作计划中不存在，则视为新增标的。
- 新增标的当天 `estimatedImpactAmount` 记为空或 0，并标记原因 `新增标的，当日不计入`。
- 从下一个复盘日开始，只要该标的仍在同一账户计划中，就可以按复盘日涨跌纳入估算。

使用这种模式时，界面必须标记为 `账户估算影响` 或 `账户影响`，不能叫 `真实盈亏`。详情页和列表应复用同一套计算结果；完成复盘后优先展示固化结果。

列表页不展示精确估算值，例如 `+1.8%` 或 `-0.4%`。列表只展示区间档位，例如：

```text
-3~-2%
-1~0%
≈0%
0~1%
1~2%
```

精确计算值只作为内部映射使用，不在列表直接暴露。

## 历史数据处理

历史复盘数据不做强制回填，不伪造账户盈亏。

建议分三档处理：

### 0. 2026-05-08 及之后的可回填复盘日

用户判断 `2026-05-08` 以及之后的复盘日已经可以补算账户盈亏。实现时可以单独做一个临时回填任务：

- 扫描 `archive_date >= '2026-05-08'` 的复盘日。
- 为每个复盘日生成或更新一条 `review_account_snapshots`。
- 快照内部按结构化账户数据保存多个账户。
- 能计算真实盈亏的账户写入 `dailyPnl` 和 `dailyPnlPercent`。
- 暂时无法计算的账户保留总资产、可用资金等可确认字段，并将盈亏字段置空。
- 回填任务必须可重复执行，按 `archive_date` upsert，不生成重复记录。

这个任务是一次性历史整理，不代表所有更早历史日期都可以被自动推算。

### 1. 已有账户化操作计划的历史复盘

如果历史复盘日已经有账户化操作计划，并且标的存在复盘日价格涨跌数据，则可以按“账户估算影响”规则即时计算或后端查询时计算。

这类历史行展示：

```text
估算影响
老虎-美股        1~2%
东方财富-国内    暂无数据
天天基金-国内    ≈0%
```

### 2. 只有旧操作计划或只有标的摘要的历史复盘

如果历史复盘只有旧的 `market_type`、`asset_plan` 文本或 `action_plan_summary` 拼接结果，但没有可靠 `account_id`，则不自动猜测账户。

这类历史行展示：

```text
估算影响
暂无数据
```

可以在详情页或未来的数据整理工具中提示用户补全账户归属，但列表页不应该把 `美股` 自动当成 `老虎-美股` 后直接算历史影响，除非该复盘日的数据已经明确迁移到账户。

### 3. 未来存在账户每日快照的复盘

如果某个历史复盘日补录了账户快照并包含固化 `impact`，则列表和详情页优先展示固化账户影响。

固化结果优先级高于即时估算：

```text
固化账户影响 > 当前即时估算 > 已记录快照 > 暂无数据
```

### 历史展示原则

- 不为历史数据批量生成假的账户盈亏。
- 不因为缺少账户估算影响而隐藏历史复盘行。
- 历史行可以显示 `暂无数据`，但仍保留 `当日主线`、`查看`、`编辑`。
- 如果只有部分账户能算，则只展示可计算账户，其他账户显示中性缺失状态或省略。
- 任何估算结果都必须明确标记为 `估算影响`。

## 视觉规则

- 账户名保持中性色。
- 涨跌方向色只用于数值、细条、热力背景或小型方向提示。
- 不保留大面积状态列。
- `编辑` 按钮视觉权重高于 `查看`。
- `查看` 保留，但作为次要按钮。
- 常见桌面宽度下，右侧列不能被裁掉或隐藏。
- 窄屏下可以横向滚动或折叠次要文本，但操作按钮必须保持对齐。

## 待确认问题

- 是否现在就新增真实账户每日快照，还是先用估算影响完成列表改造？
