# 结构化操作计划前端样式探索

## 背景

当前每日复盘中的“个股与资产操作计划”仍然是 `asset_plan` 文本字段。随着每日记录越来越多，纯文本不利于快速扫描、按标的回看、对比计划，也容易把开仓、止盈、止损、仓位和思考混在一起。

本次探索先不进入完整实现，第一步只做前端样式预览，让实际使用感先跑通。

## 已确认方向

操作计划采用主子表关系：

```text
daily_review_archive
  archive_date = 2026-05-07

daily_review_action_plans
  archive_date = 2026-05-07, symbol = MU
  archive_date = 2026-05-07, symbol = MSFT
  archive_date = 2026-05-07, symbol = GOOGL
```

历史记录通过 `archive_date` 保留：

```text
2026-05-06 + MU
2026-05-07 + MU
2026-05-08 + MU
```

因此不需要为这个需求单独设计快照表。每日复盘主表负责“某一天的复盘”，操作计划子表负责“这一天下面每个标的的一行结构化计划”。

## 子表唯一性

第一版按“一天一个标的一行完整计划”设计：

```sql
UNIQUE(archive_date, symbol)
```

这符合当前使用习惯：每个标的一行，行内包含动作、开仓计划、止盈计划、止损计划、支撑压力位、当前仓位和思考。

## 已确认字段

前端表头：

```text
| 标的 | 动作 | 开仓计划 | 止盈计划 | 止损计划 | 支撑压力位 | 当前仓位 | 思考 |
```

后端字段草案：

```text
archive_date
symbol
action_type
entry_plan
take_profit_plan
stop_loss_plan
key_levels
current_position
thinking
sort_order
created_at
updated_at
```

字段含义：

| 字段 | 展示名 | 用途 |
| --- | --- | --- |
| `symbol` | 标的 | 股票、ETF、指数、商品或现金项；前端只显示代码，不显示中文名 |
| `action_type` | 动作 | 固定三态：准备开仓、持仓观察、已清仓复盘 |
| `entry_plan` | 开仓计划 | 进入条件、分批方式、目标仓位 |
| `take_profit_plan` | 止盈计划 | 目标价、分批止盈、移动止盈规则 |
| `stop_loss_plan` | 止损计划 | 跌破位置、亏损比例、逻辑失效条件 |
| `key_levels` | 支撑压力位 | 多行结构化文本，保留支撑/压力、区间强弱和备注 |
| `current_position` | 当前仓位 | 固定四档：`0-10%`、`10%-20%`、`20%-30%`、`>30%` |
| `thinking` | 思考 | 操作理由、风险点、备选剧本 |

### 动作枚举

```text
准备开仓
持仓观察
已清仓复盘
```

### 当前仓位枚举

```text
0-10%
10%-20%
20%-30%
>30%
```

### 支撑压力位格式

支撑压力位不拆成多张细表，也不拆成过多字段。原因是这里既包含价格区间，也包含强弱、备注和临盘判断，强行拆字段会降低填写效率。

第一版使用一个多行文本字段 `key_levels`，前端表格中压缩预览，编辑区完整输入。

推荐格式：

```text
支撑位 (Support):
335-348（中）
298-310（小）
272-284（中）
246-264（强）
228-240（强） -> 突破看前高
210-222（强） -> 突破止跌
175-187（强）

压力位:
335-348（中）
```

## 第一阶段目标：只出前端样式

第一步只做一个可视化样式预览，不接真实保存逻辑，不改数据库，不改 Worker API。

当前测试页已经生成在：

```text
tests/structured_action_plan_style.html
```

样式目标：

- 模拟真实每日复盘抽屉，而不是孤立组件。
- 在复盘抽屉的“个股与资产操作计划”步骤中，用结构化表格替代大 textarea。
- 表格第一眼要能快速扫出：哪些标的要动、怎么动、风险在哪里。
- 每行默认紧凑展示，长内容做截断或多行压缩。
- 点击某行后展开编辑区，避免表格横向过宽导致难用。
- 动作用选择控件展示，不再手写。
- 当前仓位用固定枚举展示。
- 支撑压力位、止盈、止损要在视觉上靠近，方便形成交易闭环。

## 前端方案评估结论

测试页中生成过三个版本：

| 版本 | 形态 | 结论 |
| --- | --- | --- |
| 版本 A | 表格主视图 + 下方行内编辑 | 选为第一版默认方案 |
| 版本 B | 左侧标的列表 + 右侧详情编辑 | 支撑长文本更舒服，但与现有复盘步骤差异更大 |
| 版本 C | 卡片式扫描 | 适合阅读，但信息密度偏低 |

最终选择版本 A。

原因：

- 与当前复盘抽屉的改造成本最低。
- 表格能快速扫全局。
- 下方编辑区可以承载支撑压力位这种长文本。
- 不会把真实复盘页改成完全不同的信息架构。

## 后续实现范围

样式确认后再进入实现阶段，届时保存逻辑应遵循：

```text
保存某天复盘：
1. upsert daily_review_archive 主表
2. upsert daily_review_action_plans 子表
3. 删除本次提交里已经不存在的当前 archive_date 子计划
4. 删除范围只能限定在当前 archive_date，不能影响其他历史日期
```

`asset_plan TEXT` 可以继续保留为兼容摘要字段，由结构化计划生成 Markdown 摘要，供旧列表、旧导出和过渡期展示使用。

## 建议的 OpenSpec 变更

建议新建 change：

```text
structure-review-action-plans
```

建议 scope：

- 新增 `daily_review_action_plans` 子表。
- Worker `GET /api/reviews/:date/bootstrap` 返回 `actionPlans`。
- Worker 保存复盘时接收 `actionPlans` 并同步子表。
- 前端将操作计划步骤替换为版本 A。
- 旧 `asset_plan TEXT` 保留为兼容摘要字段。
- 旧历史复盘没有结构化数据时，先回退显示旧 `asset_plan` 文本。

第一版不做：

- 不做快照表。
- 不做支撑压力位细字段拆分。
- 不做自动把旧文本解析成结构化计划。
- 不改变其他复盘步骤。
