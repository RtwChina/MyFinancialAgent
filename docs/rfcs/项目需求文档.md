# 项目需求文档 (PRD)：个人股票自动化复盘系统 v3.0

## 1. 项目概述

### 1.1 项目背景

当前项目已经具备以下能力： 

-  自动抓取指定标的的价格数据与财经新闻
- 将数据写入本地 SQLite 数据库
- 导出 Excel 作为调试和人工查看结果
- 初步支持新闻筛选、AI 总结和复盘内容存档

下一阶段要将系统从“本地脚本 + 本地数据库”升级为“Cloudflare 云端数据底座 + 私有复盘网页”：

- 采集结果不再以本地 SQLite 作为主存储，而是以 Cloudflare D1 作为主数据库
- 在 Cloudflare 上提供一个网页，支持查询复盘数据
- 在 Cloudflare 上提供一个网页，支持对待复盘日期执行复盘、回填源数据并填写自定义内容

### 1.2 本阶段目标

本阶段聚焦以下 3 个结果：

1. 将价格、新闻、新闻分析、复盘数据统一沉淀到 Cloudflare D1
2. 提供一个仅自己使用的复盘网页
3. 让“待复盘日期”的生成、回填、编辑、保存形成完整闭环

### 1.3 目标用户

- 单用户场景，主要使用者是项目作者本人
- 优先桌面端使用体验
- 页面默认私有访问，不考虑公开访问场景

### 1.4 本阶段范围

本阶段包含：

- Python 采集脚本继续保留
- 数据主存储切换为 Cloudflare D1
- 新增 Cloudflare 网页，用于查询和复盘
- 明确待复盘日期逻辑、回填逻辑、保存逻辑

本阶段暂不包含：

- 多用户协作
- 完整权限系统
- 移动端独立适配
- 对外公开 API 平台化

## 3. 目标架构

### 3.1 推荐架构

| 层级 | 技术方案 | 说明 |
| --- | --- | --- |
| 数据采集 | Python | 继续负责抓取价格、新闻、LLM 分析 |
| 调度 | GitHub Actions | 定时执行采集脚本 |
| 前端页面 | Cloudflare Pages | 承载复盘网页 |
| 后端 API | Cloudflare Workers + Hono | 提供查询、回填、保存接口 |
| 数据库 | Cloudflare D1 | 作为主数据库 |
| 访问控制 | Cloudflare Access 或等效私有访问方案 | 页面默认仅自己可访问 |

### 3.2 推荐数据流

1. GitHub Actions 定时触发 Python 采集脚本
2. Python 脚本完成价格、新闻、分析数据抓取
3. 采集结果通过 Cloudflare Workers API 写入 D1
4. Cloudflare Pages 上的网页通过 Workers API 查询 D1
5. 用户在网页中打开待复盘日期，系统自动回填源数据
6. 用户编辑复盘内容并保存草稿或标记完成

### 3.3 存储策略

- Cloudflare D1 为生产主数据源
- 本地 SQLite 仅作为开发调试或离线兜底能力
- Excel 导出保留为可选调试输出，不作为正式数据源

## 4. 核心业务流程

### 4.1 自动采集流程

#### 价格采集

- 每天在美股收盘后运行一次
- 获取最近一个已收盘交易日的价格数据
- 写入 D1 的 `stock_raw`
- 可选导出 Excel

#### 新闻采集

- 每 6 小时执行一次
- 先抓取多源原始新闻，再执行规则初筛
- 规则初筛只保留两类新闻：
  - 能明显影响全球经济或美股大盘的重大新闻
  - 与跟踪标的直接相关、且可能影响交易判断的标的新闻
- 明显噪音新闻直接丢弃，不进入正式新闻库
- 初筛通过后，按批次送入 LLM 做进一步精选与结构化概述
- LLM 输出写回 `news_raw_data` 的增强字段
- 批次级和日期级综合分析写入 `daily_news_ai_analysis`

#### 新闻处理分层流程

新闻处理改为 3 层：

1. 原始抓取层
- 从新浪、财联社、金十、Yahoo Finance 等来源抓取原始新闻
- 统一基础字段：`pub_date`、`title`、`content`、`url`、`source`
- 对重复新闻做去重

2. 规则初筛层
- 目标不是“尽量多收”，而是优先保留真正可能影响复盘和交易判断的新闻
- 规则初筛通过的新闻才允许进入正式 `news_raw_data`
- 初筛应输出明确的规则结果，例如：
  - 是否通过初筛
  - 初筛类别：`index` / `sector` / `stock`
  - 初筛原因
  - 规则分数

3. LLM 精选层
- 只处理已通过规则初筛的新闻
- 不对每条新闻单独发请求，而是按批次聚合后并发请求
- LLM 需要返回结构化结果，用于补全单条新闻摘要、市场影响、重要度、归属标的
- 每轮 6 小时采集任务结束后，都要围绕最近一个已收盘交易日 `T` 做一次日期级综合分析重算，并覆盖写入 `daily_news_ai_analysis`

### 4.2 待复盘日期生成流程

待复盘日期是本项目的关键对象。

定义规则：

- 以美股交易日为主日历
- 对于每一个已收盘的有效交易日，如果该日期没有完成复盘，则视为“待复盘日期”
- 如果该日期已有复盘记录但状态为 `draft`，也视为待复盘
- 如果状态为 `completed`，则不再出现在待复盘列表

待复盘日期的生成逻辑：

1. 根据当前时间和美股交易日历计算最近一个已收盘交易日 `T`
2. 获取截至 `T` 的最近若干个有效交易日
3. 从 `daily_review_archive` 中查询这些日期的复盘状态
4. 输出状态为“未复盘”或“草稿中”的日期列表

### 4.3 执行复盘流程

当用户点击某个待复盘日期后，系统需要自动完成以下回填动作：

1. 计算该复盘日期对应的新闻窗口：
   - 起点：`T-2` 交易日收盘时刻
   - 终点：`T` 交易日收盘时刻
2. 查询该复盘日期 `T` 的价格数据
3. 查询上述时间窗口内的新闻数据
4. 查询该日期对应的新闻分析结果
5. 查询前一个已完成复盘日期的以下字段作为默认参考值：
   - 大盘流动性追踪
   - 大宗商品与板块轮动
   - 个股与资产操作计划
6. 将上述内容回填到复盘页面
7. 用户可继续补充和修改自定义内容
8. 用户可选择“保存草稿”或“标记完成”

## 5. 网页功能需求

网页当前按 3 个核心页面设计：

1. 新闻总览页 `News Explorer`
2. 复盘工作台页 `Review Workspace`
3. 标的管理页 `Symbol Manager`

### 5.1 页面一：新闻总览页 `News Explorer`

#### 页面目标

支持查看当前数据库里存储的所有新闻，完成搜索、筛选、归属标的判断，并进入新闻详情。

#### 页面能力

- 查看所有已入库新闻
- 按关键词搜索标题、摘要、正文
- 按日期范围筛选
- 按来源筛选：`sina / cls_cn / jin10 / yahoo_finance`
- 按新闻类型筛选：`大盘新闻 / 板块新闻 / 个股新闻`
- 按标的筛选：`MU / LITE / MSFT / GOOGL / VIX ...`
- 区分“是否已归属于具体标的”
- 区分“是否通过规则初筛”
- 点击查看新闻详情

#### 建议页面结构

1. 顶部搜索与筛选条
2. 中部新闻列表区
3. 右侧详情抽屉，或点击后打开详情弹层

#### 页面布局建议

1. 顶部筛选条
- 关键词搜索框
- 日期范围选择
- 来源筛选
- 新闻类型筛选
- 标的筛选
- 是否已归属标的筛选

2. 新闻列表区
- 采用表格或卡片列表
- 默认按发布时间倒序
- 支持分页或无限滚动

3. 新闻详情区
- 点击单条新闻后展开
- 显示全文、AI 摘要、归属标的信息、原始链接

#### 列表字段建议

| 字段 | 说明 |
| --- | --- |
| 发布时间 | `pub_date` |
| 标题 | `title` |
| 来源 | `source` |
| 新闻类型 | `type` |
| 主归属标的 | `primary_symbol` |
| 归属标的标签 | `related_symbols` |
| AI 摘要预览 | `ai_summary` |
| 是否纳入复盘候选 | `is_relevant_to_review` |
| 操作 | 查看详情 |

#### 详情区展示内容

- 发布时间
- 标题
- 来源
- 原始正文 / 摘要
- AI 摘要
- 主归属标的
- 归属标的列表
- 是否建议纳入复盘
- 原始链接
- 标的归属判断依据（如后续需要，可单独展示或写入标签）

#### 页面设计要点

- 必须让用户一眼看出“这条新闻是否归属于某个标的”
- 标的归属支持一对多，一条新闻可以关联多个标的
- 搜索结果应支持与筛选条件叠加
- 列表区要足够轻，不要默认直接塞入长正文

### 5.2 页面二：复盘工作台 `Review Workspace`

#### 页面目标

将“待复盘日期”和“全量复盘记录”放到同一个工作台中，但将复盘编辑区改为点击后弹出或跳转，避免当前页面结构过重。

#### 页面能力

- 顶部横向展示待复盘日期
- 点击 `开始复盘` 后打开复盘编辑区
- 编辑区自动回填价格、新闻、新闻分析、前一日参考字段
- 支持保存草稿
- 支持标记完成
- 页面下半部分始终展示全量复盘列表
- 已完成复盘后能回到全量列表查看结果

#### 页面结构建议

1. 顶部横向待复盘日期条
   - 横向卡片式展示待复盘日期
   - 每个卡片显示：日期、状态、`开始复盘` 按钮
   - 如果日期较多，采用横向滚动
   - 默认展示最近待处理日期

2. 中部复盘编辑区
   - 不建议默认常驻展示
   - 用户点击 `开始复盘` 后再打开
   - 形式可为全屏抽屉、弹层或独立编辑页

3. 下部全量复盘列表
   - 始终保留
   - 展示所有历史复盘记录
   - 支持查看、继续编辑、筛选状态

#### 顶部横向待复盘日期条字段建议

| 字段 | 说明 |
| --- | --- |
| 复盘日期 | `archive_date` |
| 当前状态 | `missing / draft` |
| 最近更新时间 | `updated_at` |
| 操作 | `开始复盘` |

#### 复盘编辑区展示区块

1. 顶部摘要区
   - 复盘日期
   - 复盘状态
   - 新闻窗口
   - 上次保存时间
   - 保存草稿
   - 标记完成

2. 源数据回填区（三段式折叠）
   - **大盘分析**：含指数/商品/汇率价格表 + AI 大盘解读（从 `sector_impact_map` 的 [大盘] 标记提取）
   - **板块分析**：含板块 ETF 价格表 + AI 板块解读（从 `sector_impact_map` 的 [板块] 标记提取）
   - **个股深度研究**：含个股价格表 + AI 个股解读（从 `sector_impact_map` 的 [个股] 标记提取）
   - 每段可折叠，折叠时标题行显示关键价格变动摘要（如 `标普 -0.8% · VIX +5.2%`）
   - 折叠状态存 localStorage，下次打开复盘记住
   - 新闻总结步骤显示每日总结（`daily_major_events`）+ 逻辑链（`linkage_logic_chain`）

3. 人工编辑区
   - 历史价位复盘
   - 新闻总结
   - 大盘流动性追踪
   - 大宗商品与板块轮动
   - 个股与资产操作计划
   - 自定义补充内容
   - 深度思考与交易总结

#### 全量复盘列表字段建议

| 字段 | 说明 |
| --- | --- |
| 复盘日期 | `archive_date` |
| 复盘状态 | `draft / completed` |
| 最后更新时间 | `updated_at` |
| 是否存在源数据快照 | `source_snapshot_json` 是否为空 |
| 操作 | 查看 / 继续编辑 |

#### 页面交互要求

- 待复盘日期必须横向展示，不再采用左侧竖向列表
- 点击 `开始复盘` 后再展示复盘编辑区
- 如果仅有一个待复盘日期，可默认聚焦该日期卡片
- 如果有多个待复盘日期，用户先点击某个日期再进入编辑
- 保存草稿后，保留在待复盘列表中
- 标记完成后，从待复盘列表中移除
- 标记完成后，页面下方全量复盘列表应立即出现或更新对应记录
- 复盘工作台首页的重点是“待处理入口 + 全量列表”，不是默认塞满编辑表单

### 5.3 页面三：标的管理 `Symbol Manager`

#### 页面目标

维护系统跟踪的全部金融标的，是新闻分类、价格采集、复盘分组的唯一权威数据源。

#### 页面能力

- 按类型（大盘/板块/个股）分组展示标的列表
- 智能添加：输入名称或代码，AI 自动识别并补全 Yahoo 代码、别名、类型
- 手动添加：直接填写结构化表单
- 编辑标的：修改显示名称、Yahoo 代码、别名、类型
- 删除标的（软删除，`is_active=0`）
- 拖拽调整同类型内的显示顺序（`sort_order`）

#### 数据字段说明

| 字段 | 说明 |
| --- | --- |
| 系统代码 `symbol` | 系统内唯一标识，人类友好（如 `GSPC`、`VIX`、`MU`） |
| Yahoo 代码 `yahoo_symbol` | Yahoo Finance 采集用代码（如 `^GSPC`、`GC=F`），仅价格采集时使用 |
| 显示名称 `display_name` | 中文展示名（如 标普500、恐慌指数） |
| 类型 `symbol_type` | `index`（大盘/指数/大宗商品）/ `sector`（板块/ETF）/ `stock`（个股） |
| 别名 `aliases` | JSON 数组，用于新闻文章匹配，只含文章中实际出现的写法，**不含 Yahoo 技术代码** |
| 排序 `sort_order` | 同类型内拖拽排序值 |

#### 设计要点

- `symbol` 和 `yahoo_symbol` 必须分离：前者是系统语言，后者是采集层映射
- `aliases` 只存新闻里出现的文字模式，`^GSPC`、`GC=F` 这类技术代码**不应在 aliases 中出现**
- 智能识别（LLM 解析 + Yahoo 验价）和手动填写是平行入口
- 标的类型决定其在复盘价格区的分组归属

## 6. 数据与表结构需求

### 6.1 数据原则

- 所有生产数据最终写入 Cloudflare D1
- 写入必须支持幂等，避免重复插入
- 新闻数据持续积累，不做删除
- 复盘记录允许多次编辑，但同一复盘日期最终只保留一条主记录

### 6.2 表 A：`stock_raw`（原始价格表）

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `id` | INTEGER PRIMARY KEY | 主键 |
| `k_date` | TEXT NOT NULL | 交易日期，格式 `YYYY-MM-DD` |
| `stock_code` | TEXT | 股票代码 |
| `stock_name` | TEXT | 股票名称 |
| `symbol` | TEXT NOT NULL | 交易符号 |
| `current_price` | REAL | 收盘价 |
| `change_percent` | REAL | 相比前一日收盘价的涨跌幅 |
| `volume` | INTEGER | 成交量 |
| `captured_at` | DATETIME | 采集时间 |
| `created_at` | DATETIME DEFAULT CURRENT_TIMESTAMP | 创建时间 |

唯一约束：

- `UNIQUE(k_date, symbol)`

### 6.3 表 B：`news_raw_data`（原始新闻表）

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `id` | INTEGER PRIMARY KEY | 主键 |
| `pub_date` | DATETIME | 新闻原始发布时间 |
| `title` | TEXT | 标题 |
| `summary` | TEXT | 摘要 |
| `content` | TEXT | 正文 |
| `url` | TEXT | 链接 |
| `source` | TEXT | 来源，例如 `sina`、`cls_cn`、`jin10`、`yahoo_finance` |
| `type` | TEXT | 类型，取值：`index` / `sector` / `stock`（旧值 `macro`/`market`/`symbol` 已通过 Migration 迁移） |
| `rule_passed` | INTEGER | 是否通过规则初筛，0/1 |
| `rule_score` | REAL | 规则初筛分数 |
| `rule_reason` | TEXT | 规则通过原因或保留原因 |
| `ai_summary` | TEXT | 单条新闻 AI 摘要 |
| `market_impact` | TEXT | LLM 给出的市场影响概述 |
| `importance_level` | TEXT | 重要度，建议取值：`high` / `medium` / `low` |
| `primary_symbol` | TEXT | 主归属标的 |
| `related_symbols` | TEXT | 归属标的列表，建议保存为 JSON 字符串 |
| `is_relevant_to_review` | INTEGER | 是否建议纳入复盘候选，0/1 |
| `llm_batch_id` | TEXT | 本条新闻所属 LLM 批次标识 |
| `news_hash` | TEXT | 去重唯一标识 |
| `captured_at` | DATETIME | 采集时间 |
| `created_at` | DATETIME DEFAULT CURRENT_TIMESTAMP | 创建时间 |

唯一约束：

- `UNIQUE(news_hash)`

设计说明：

- `news_raw_data` 不再保存“所有抓到的原始新闻”，而是保存“通过规则初筛，并经过 LLM 结构化增强后的有效新闻”
- `rule_passed` / `rule_score` / `rule_reason` 用于保留规则层判断结果，方便后续调试和回看
- `market_impact` 用于沉淀“这条新闻为什么影响市场”的结构化说明
- `primary_symbol` 用于在新闻列表页直观展示“这条新闻主要归属哪个标的”
- `related_symbols` 用于支持一条新闻归属多个标的
- `is_relevant_to_review` 用于在新闻页和复盘页快速筛选“建议纳入复盘”的新闻

### 6.4 表 C：`daily_news_ai_analysis`（新闻分析结果表）

| 字段名 | 类型 | 说明 |
| --- | --- | --- |

| `batch_no` | INTEGER | LLM 批次编号，日期级汇总可为空 |
| `analysis_scope` | TEXT | 分析范围，建议取值：`batch` / `daily_summary` |
| `global_news` | TEXT | 全球重大新闻 |
| `market_news` | TEXT | 股票市场重大新闻 |
| `symbol_news` | TEXT | 标的相关新闻摘要 |
| `market_analysis` | TEXT | 市场分析摘要 |
| `raw_summary` | TEXT | 完整分析输出 |
| `created_at` | DATETIME DEFAULT CURRENT_TIMESTAMP | 创建时间 |

建议索引：

- `INDEX analysis_date`

设计说明：

- `daily_news_ai_analysis` 既要支持保存单个 LLM 批次的原始输出，也要支持保存交易日维度的日期级综合分析

## 8. 采集任务需求

### 8.1 价格采集任务

- 根据交易日锚定逻辑计算最近一个已收盘交易日
- 获取指定标的价格
- 写入 D1 的 `stock_raw`
- 默认支持幂等
- 保留日志
- 可选保留 Excel 导出

### 8.2 新闻采集任务

- 抓取多源原始新闻
- 生成新闻唯一标识并去重
- 先执行规则初筛，丢弃明显无关新闻
- 对通过初筛的新闻分配 `type`、`rule_score`、`rule_reason`
- 将初筛通过新闻按固定批次聚合，送入 LLM 并发处理
- 将 LLM 返回的单条新闻摘要、市场影响、重要度、归属标的写回 `news_raw_data`
- 将批次分析结果写入 `daily_news_ai_analysis`
- 每轮任务结束后，基于数据库中属于交易日 `T` 复盘窗口的全部有效新闻，重算一次 `daily_summary` 级别的日期综合分析，并以 upsert 方式覆盖写入 `daily_news_ai_analysis`
- 保留日志
- 可选保留 Excel 导出

#### 8.2.1 规则初筛要求

规则初筛必须尽量稳定、可解释，并优先解决“把杂讯挡在库外面”。

应保留的新闻类型：

- 宏观与全球经济类：
  - 美联储、利率、通胀、非农、就业、CPI、PPI
  - 关税、制裁、重大政策变化、财政刺激、流动性事件
  - 战争、重大地缘政治冲突、能源运输中断、重要自然灾害
- 股市大盘与市场结构类：
  - 标普/纳指/道指显著波动原因
  - 重要财报、盈利预警、监管处罚、并购、回购、分红、IPO
  - 对科技、半导体、AI、金融、能源等关键板块有明显影响的事件
- 标的相关类：
  - 明确提到跟踪标的或其公司名
  - 明确影响跟踪标的所在行业链条的新闻
  - 跟踪标的财报、指引、产品、监管、诉讼、收购、重大合作

应优先丢弃的新闻类型：

- 纯情绪化点评、泛泛而谈的市场评论
- 分析师评级、目标价调整、缺乏事件增量的信息
- 盘中短促噪音、重复快讯、无明确市场含义的碎片化播报
- 与全球经济、美股大盘、跟踪标的均无明显关系的内容

#### 8.2.2 LLM 精选要求

- LLM 只处理规则初筛通过的新闻
- 每次请求处理一个新闻批次，而不是单条请求
- 批次大小建议控制在 `8-12` 条新闻
- 并发批次数建议控制在 `2-3`
- LLM 必须返回严格结构化结果，至少包含：
  - `news_hash`
  - `keep` 是否保留
  - `ai_summary`
  - `market_impact`
  - `importance_level`
  - `type`
  - `primary_symbol`
  - `related_symbols`
- 如果某条新闻在 LLM 层被判定为不重要，可直接标记为 `keep=false`，不写入正式新闻库
- 所有批次处理完成后，再基于最终保留新闻生成一次日期级综合分析

#### 8.2.3 LLM 压力控制要求

- 不允许对每条新闻逐条发起 LLM 请求
- 必须先规则初筛，再进入 LLM，减少 token 压力
- 必须使用统一模板格式化批次输入，避免模型输出漂移
- 批次内每条新闻必须带稳定标识，例如 `news_hash`
- 对 LLM 输出必须做解析校验；格式不合法时允许重试或降级
- 当 LLM 失败时，允许保留规则初筛结果并跳过增强字段，但要记录失败日志

#### 8.2.4 日期级综合分析生成逻辑

- 交易日锚点使用最近一个已收盘美股交易日 `T`
- 每次 6 小时新闻任务结束后，不直接新增一条“最终分析”，而是执行一次“针对 `T` 的滚动重算”
- 重算数据范围为 `T-2_close` 到 `T_close` 窗口内、已经通过规则初筛且最终保留的新闻
- 该重算结果写入 `daily_news_ai_analysis` 中 `analysis_scope = 'daily_summary'` 的记录
- 同一个 `analysis_date = T` 只保留一条 `daily_summary` 记录，后续任务以 upsert 覆盖更新
- 这样复盘页始终读取的是交易日 `T` 的最新综合结论，而不是同一天内多条相互冲突的“最终分析”

### 8.3 输出策略调整

从本阶段开始：

- D1 是主输出目标
- 本地 SQLite 不再作为生产主库
- 如需保留本地 SQLite，仅用于开发调试

## 10. 交易日锚定与复盘时间窗口

### 10.1 核心定义

- `now`：脚本实际执行时间
- `T`：距离 `now` 最近一个已完成收盘的美股交易日
- `T-2`：相对 `T` 往前推 2 个有效交易日
- 新闻窗口：`T-2` 收盘时刻到 `T` 收盘时刻

### 10.2 复盘时间窗口规则

示例：

- 如果在周二凌晨执行，`T` 通常为周一，`T-2` 为上周四
- 如果在周六凌晨执行，`T` 通常为周五，`T-2` 为本周三

要求：

- 所有时间必须明确时区
- 至少标明北京时间和美东时间换算逻辑
- 必须正确处理夏令时、周末和节假日

### 10.3 复盘页使用该窗口的方式

复盘页针对日期 `T` 自动加载：

- `stock_raw` 中 `k_date = T` 的价格数据
- `news_raw_data` 中 `pub_date` 位于 `T-2_close` 至 `T_close` 的新闻
- `daily_news_ai_analysis` 中 `analysis_date = T` 的分析结果

## 11. 非功能要求

### 11.1 数据一致性

- 重复执行采集任务不能产生重复脏数据
- 同一复盘日期只能有一条主记录
- 页面保存后再次打开应能恢复最新草稿
- 新闻规则初筛和 LLM 精选结果必须可追溯，不能只保留最终结论而丢失判断依据

### 11.2 可维护性

- 所有敏感信息必须通过环境变量或平台 Secret 管理
- 不得在需求文档和代码中写死真实 API Key
- 日志要能区分采集失败、入库失败、页面保存失败

### 11.3 页面体验

- 优先桌面端使用
- 查询常用接口响应时间尽量控制在 3 秒内
- 复盘详情页信息区块要清晰，不要把所有数据堆成单一长文本
