# 项目需求文档 (PRD)：个人股票自动化复盘系统 v3.1

> 最后更新: 2026-03-18
> 状态: v3.0 已基本实现，本文档为实现后的对齐修订版

## 1. 项目概述

### 1.1 项目背景

本项目是一套面向个人投资者的股票自动化复盘系统，已完成从"本地脚本 + SQLite"到"Cloudflare 云端数据底座 + 私有复盘网页"的升级：

- 自动采集指定标的的价格数据与多源财经新闻
- 通过规则初筛 + LLM 精选的多层新闻过滤，筛选真正有价值的新闻
- 数据持久化到 Cloudflare D1（云端 SQLite）
- 提供复盘工作台网页，支持查询、编辑、归档

### 1.2 当前实现状态

| 模块 | 状态 | 说明 |
|------|------|------|
| 价格采集 | ✅ 已完成 | yfinance 采集 → Cloudflare D1，支持 40 个标的 |
| 新闻采集 | ✅ 已完成 | 4 源采集（新浪/财联社/金十/Yahoo）→ 规则初筛 → LLM 精选 → D1 |
| AI 新闻分析 | ✅ 已完成 | 批次级 + 日期级综合分析，upsert 覆盖更新 |
| 云端数据库 | ✅ 已完成 | Cloudflare D1，6 张表，7 次迁移 |
| Worker API | ✅ 已完成 | 写入接口（鉴权）+ 查询接口 + 复盘接口 |
| 复盘网页 | ✅ 已完成 | 新闻总览 + 复盘工作台（草稿/完成/回填） |
| 标的管理 | ✅ 已完成 | tracked_symbols 表，40 个活跃标的 |
| 定时调度 | ✅ 已完成 | GitHub Actions（新闻每小时 + 收盘后价格汇总） |
| 集成测试 | ✅ 已完成 | replay fixture 回放测试框架 |

### 1.3 目标用户

- 单用户场景，主要使用者是项目作者本人
- 优先桌面端使用体验
- 页面默认私有访问，不考虑公开访问场景

### 1.4 项目范围

**已实现：**
- Python 采集端（价格 + 新闻 + LLM 分析）
- Cloudflare D1 作为主数据库
- Cloudflare Worker API（写入 + 查询 + 复盘管理）
- 复盘工作台网页（新闻浏览 + 复盘编辑）
- GitHub Actions 定时调度
- 标的主数据管理（tracked_symbols）
- 集成测试框架（数据源重放）

**暂不包含：**
- 多用户协作
- 完整权限系统（当前仅 Bearer Token 鉴权写入接口）
- 移动端独立适配
- 对外公开 API 平台化

## 2. 技术架构

### 2.1 整体架构

| 层级 | 技术方案 | 说明 |
|------|----------|------|
| 数据采集 | Python 3.12 | 价格、新闻抓取，LLM 分析 |
| 调度 | GitHub Actions | 两个定时 Workflow |
| 前端页面 | 原生 HTML/CSS/JS | 通过 Worker 提供静态资源 |
| 后端 API | Cloudflare Workers (JS) | 单文件路由，处理读写 |
| 数据库 | Cloudflare D1 | 主数据库，SQLite 兼容 |
| 访问控制 | Bearer Token | 写入接口鉴权 |

### 2.2 数据流

1. GitHub Actions 定时触发 Python 采集脚本
2. 价格采集：yfinance → `POST /api/ingest/prices` → D1 `stock_raw`
3. 新闻采集：4 源抓取 → 规则初筛 → LLM 批量精选 → `POST /api/ingest/news` → D1 `news_raw_data`
4. AI 分析：日期级综合分析 → `POST /api/ingest/news-analysis` → D1 `daily_news_ai_analysis`
5. Web 前端通过 Worker API 查询 D1 展示数据
6. 用户在复盘工作台编辑并保存复盘内容

### 2.3 环境部署

| 环境 | 分支 | Worker | D1 数据库 |
|------|------|--------|-----------|
| 生产 | `main` | `my-financial-agent.rtw1994.workers.dev` | `my-financial-agent` |
| 测试 | `test` | `my-financial-agent-test.rtw1994.workers.dev` | `my-financial-agent-test` |

## 3. 核心业务流程

### 3.1 自动采集流程

#### 价格采集（close-summary 模式）

- 触发：每天北京时间 05:00（美股收盘后约 30 分钟）
- 流程：
  1. 根据 NYSE 交易日历计算最近已收盘交易日 `T`
  2. 通过 yfinance 获取 tracked_symbols 中所有活跃标的的收盘价
  3. 补采最新新闻并写入 D1
  4. 生成日期级 AI 综合分析
  5. 初始化复盘记录

#### 新闻采集（full 模式）

- 触发：每小时执行一次
- 流程：
  1. 从新浪、财联社、金十、Yahoo Finance 并发抓取新闻
  2. 生成 news_hash（MD5(title+source)）去重
  3. 规则初筛：LLM 判断类型（index/sector/stock）和相关性
  4. LLM 批量精选（batch_size=6, max_workers=2）：生成摘要、市场影响、重要度、归属标的
  5. 批量写入 D1
  6. 日期级综合分析（upsert 覆盖更新同一交易日的分析）
  7. 初始化复盘记录

#### 新闻处理分层

1. **原始抓取层**：统一 4 源格式，基础去重
2. **规则初筛层**：LLM 判断类型和通过/不通过，输出 rule_passed / rule_reason
3. **LLM 精选层**：按批次聚合处理，返回结构化摘要和影响分析

#### 应保留的新闻

- 宏观经济：美联储、利率、通胀、非农、CPI、关税、制裁、地缘冲突
- 市场结构：标普/纳指/道指显著波动、重要财报、并购、IPO
- 标的相关：明确提到跟踪标的的财报、产品、监管、诉讼等

#### 应丢弃的新闻

- 纯情绪化评论、无事件增量的分析师评级
- 盘中噪音、重复快讯、碎片化播报
- 与跟踪范围无关的内容

### 3.2 待复盘日期

定义规则：
- 以美股交易日为主日历
- 已收盘且未完成复盘（状态为 `missing` 或 `draft`）的日期 = 待复盘
- 状态为 `reviewed` 的日期不出现在待复盘列表

### 3.3 执行复盘流程

用户点击待复盘日期后，系统通过 bootstrap 接口自动回填：

1. 当日价格数据（`stock_raw` WHERE `k_date = T`）
2. 新闻窗口内的新闻（`T-1_close` 到 `T_close`）
3. 日期级 AI 分析结果
4. 前一复盘日的参考字段（流动性追踪、板块轮动、操作计划）
5. 已有草稿内容

用户可编辑并选择"保存草稿"或"标记完成"。

## 4. 网页功能

### 4.1 新闻总览页 (News Explorer)

**已实现功能：**
- 全量新闻列表，按发布时间倒序
- 关键词搜索（标题/内容）
- 按日期范围筛选
- 按来源筛选：sina / cls_cn / jin10 / yahoo_finance
- 按类型筛选：index / sector / stock
- 分页浏览
- 展开查看详情（AI 摘要、市场影响、归属标的、原文链接）

### 4.2 复盘工作台 (Review Workspace)

**已实现功能：**
- 顶部横向待复盘日期卡片（日期 + 状态 + 开始复盘按钮）
- 复盘编辑区（点击后展开）：
  - 源数据回填（价格表格 + AI 新闻分析 + 关联新闻列表）
  - 人工编辑区（新闻点评、流动性追踪、板块轮动、操作计划、交易总结）
  - 保存草稿 / 标记完成
- 底部全量复盘列表（查看/继续编辑）

## 5. 数据模型

### 5.1 表结构

**`stock_raw`**（价格表）

| 字段 | 类型 | 说明 |
|------|------|------|
| k_date | TEXT | 交易日期 YYYY-MM-DD |
| symbol | TEXT | 系统标识，关联 tracked_symbols.symbol |
| yahoo_symbol | TEXT | Yahoo Finance 代码（数据来源标识） |
| stock_name | TEXT | 股票名称 |
| current_price | REAL | 收盘价 |
| change_percent | REAL | 涨跌幅 |
| volume | INTEGER | 成交量 |
| UNIQUE(k_date, symbol) | | |

**`news_raw_data`**（新闻表）

| 字段 | 类型 | 说明 |
|------|------|------|
| pub_date | DATETIME | 发布时间 |
| title | TEXT | 标题 |
| content | TEXT | 正文 |
| source | TEXT | 来源 |
| type | TEXT | 类型: index/sector/stock |
| rule_passed | INTEGER | 是否通过规则初筛 |
| rule_reason | TEXT | 规则原因 |
| processing_status | TEXT | 处理状态: rule_screened/llm_processed/llm_discarded |
| ai_summary | TEXT | AI 摘要 |
| market_impact | TEXT | 市场影响 |
| importance_stars | INTEGER | 重要度 0-5 |
| related_symbols | TEXT | 归属标的 JSON |
| news_hash | TEXT | 去重唯一标识 |
| UNIQUE(news_hash) | | |

**`daily_review_archive`**（复盘存档）

| 字段 | 类型 | 说明 |
|------|------|------|
| archive_date | TEXT PK | 复盘日期 |
| review_status | TEXT | 状态: draft/reviewed/deleted |
| reviewer_news_notes | TEXT | 新闻点评 |
| market_sentiment | TEXT | 流动性追踪 |
| sector_rotation | TEXT | 板块轮动 |
| asset_plan | TEXT | 操作计划 |
| trading_summary | TEXT | 交易总结 |

**`daily_news_ai_analysis`**（AI 分析）

| 字段 | 类型 | 说明 |
|------|------|------|
| analysis_date | TEXT PK | 分析日期 |
| daily_major_events | TEXT | 今日大事概览 |
| sector_impact_map | TEXT | 板块影响图谱 |
| linkage_logic_chain | TEXT | 联动逻辑链 |

**`daily_review_archive_news`**（复盘新闻快照）

| 字段 | 类型 | 说明 |
|------|------|------|
| archive_date | TEXT | 复盘日期 |
| 新闻字段快照 | ... | 复制自 news_raw_data |
| UNIQUE(archive_date, news_hash) | | |

**`tracked_symbols`**（标的管理）

| 字段 | 类型 | 说明 |
|------|------|------|
| symbol | TEXT UNIQUE | 系统标识 (如 GSPC, MU) |
| yahoo_symbol | TEXT | Yahoo Finance 代码 |
| display_name | TEXT | 中文显示名 |
| symbol_type | TEXT | index/sector/stock |
| aliases | TEXT | 别名 JSON 数组 |
| is_active | INTEGER | 是否启用 |
| sort_order | INTEGER | 排序 |

### 5.2 数据原则

- 所有生产数据写入 Cloudflare D1
- 写入幂等（ON CONFLICT / UNIQUE 约束）
- 新闻持续积累不删除
- 同一复盘日期只保留一条记录
- 规则初筛和 LLM 精选结果可追溯

## 9. 环境变量

### 9.1 LLM 配置

| 变量名 | 说明 | 默认值 |
|--------|------|--------|
| `LLM_API_KEY` | LLM 服务密钥 | — |
| `LLM_BASE_URL` | LLM 服务地址 | dashscope |

| `LLM_BATCH_MODEL_ID` | 批量精选模型 | qwen3.5-flash |

| `LLM_MAX_RETRIES` | 最大重试 | 2 |
| `LLM_MAX_WORKERS` | 并发数 | 2 |
| `LLM_BATCH_SIZE` | 批处理条数 | 6 |
| `SKIP_LLM` | 跳过 LLM 分析 | false |

## 10. 非功能要求

### 10.1 数据一致性

- 重复执行不产生重复数据（幂等写入）
- 同一复盘日期只有一条记录
- 新闻处理链路可追溯（rule_passed → processing_status）

### 10.2 可维护性

- 所有敏感信息通过环境变量或 Secrets 管理
- 日志区分采集失败、入库失败、页面保存失败
- 数据源通过 Protocol 抽象，可独立替换

### 10.3 页面体验

- 优先桌面端
- 暗色主题
- 复盘编辑区信息区块清晰，不堆叠长文本

## 11. 后续迭代方向
